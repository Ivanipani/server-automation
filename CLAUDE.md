# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Declarative homelab automation for **poochella**: a small Proxmox VE fleet running a k3s Kubernetes cluster on node-local LVM-thin-backed VMs, fronted by an OPNsense router. OpenTofu provisions infrastructure on Proxmox; Ansible configures everything else. The control node is the user's laptop.

**Physical hosts split by role, not by hardware.** Today: `pve-home-01` is the only Proxmox hypervisor (mini PC); `worker-home-02` (renamed from `pve-home-02`) is plain Debian 13 baremetal (gaming-PC hardware that replaced the old crash-looping unit, since de-Proxmoxed for use as a Linux worker — both VM-host and baremetal kube_worker). `pve-home-03` was retired. Inventory umbrella groups: `physical` = every baremetal Linux box (`hypervisors` ∪ `workers`); `hypervisors` is PVE-only; `workers` is plain Linux.

**Every Proxmox hypervisor is a fully standalone PVE host** — **no corosync, no shared storage, no HA, no live-migration**. The cluster code path is **gated off**: an assertion at the top of `20-hypervisor/20-cluster.yml` refuses any non-empty `pve_cluster` group, and the cluster task files under `roles/hypervisor/` are kept on disk only as deprecated reference.


## Common commands

All workflows are wrapped in `justfile` recipes — prefer `just <recipe>` over invoking `ansible-playbook` directly:

- `just check` — verify control-node deps (ansible, fzf, collections, `passlib`)
- `just install` — install ansible-core via `uv tool install` plus required collections
- `just ping` — `ansible all -m ping` smoke test
- `just run [options]` — fzf-pick a playbook under `playbooks/` and run it (auto-passes `--vault-password-file ansible-pass`)
- `just test` — fzf-pick tags from `test/test.yml` and run that subset
- `just ssh-refresh` — clear & re-seed `known_hosts` for every host in the inventory (useful after re-imaging)
- `just secret-encrypt <name>` — `ansible-vault encrypt_string` for adding values to `group_vars/all/vault.yml`
- Tofu is driven by Ansible (`13-foundation/80-tofu-infra-lxcs.yml` + `30-guests/10-opentofu.yml`); for manual per-hypervisor work, `cd tofu/node`, `tofu workspace select <host>`, and run `tofu init/plan/apply` directly — the API URL is derived from the workspace/`hypervisor_name` (`https://<host>.<domain>:8006`), so no `-var proxmox_endpoint` is needed (override only for an IP/alt port; see `tofu/README.md`). There is ONE flat Tofu root (`tofu/node/`) for the whole fleet — state per hypervisor comes from a named tofu workspace, not a per-host directory.

**Infra layout (tier directories).** `playbooks/poochella/infra/` is **seven** ordered tiers, each a directory with its own ordered `site.yml`; ordinals are gap-10 *within* a tier so steps insert without renumbering. The directory **is** the target tier:
- `10-router/` — OPNsense DNS/DHCP foundation (precedes everything)
- `15-nas/` — Synology NAS (gap-5 break; independent of host bootstrap, admin'd via DSM rather than the ansible-user path)
- `13-foundation/` — Foundational subset (the `foundation` inventory group, today just `pve-home-01`): full 17-host + 20-hypervisor stack against that host, then **bootstrap-only LOCAL LXC template bake** (`50-bootstrap-lxc-template.yml` — breaks the chicken-and-egg: bootserv01 LXC needs a Debian LXC template to exist on the foundation node before Tofu can clone it), then a **TARGETED** `tofu apply -target=module.infra_lxcs_<node>` that creates ONLY the infra LXCs (bootserv01 today) — never any VM — then the bootserv role, then the **full PRODUCER step** (`95-bake-publish-images.yml`: bake the Debian VM cloud image via `virt-customize`+`virt-sysprep` and snapshot the local LXC template via `vzdump`, then PUBLISH both to bootserv01 via `pct push` at `/srv/http/cloud-images/<image_file>` + `/srv/http/lxc-templates/<artifact>` with sha256 sidecars), then a mirror step (`96-mirror-artifacts.yml`) and the **PVE PXE producer** (`97-bake-publish-pve-pxe.yml`: bake ONE shared PVE auto-installer for the whole hypervisor fleet — patch the installer's dhclient `timeout 10`→`60` inside `pve-installer.squashfs` to beat the slow-NIC-link-up DHCP race [Proxmox bug #5583], rebuild a bootable ISO, then `prepare-iso --fetch-from http --url http://bootserv01.lan/proxmox-answer --pxe-loader ipxe`, and publish `vmlinuz`+`initrd.img`+stripped-iso flat to `/srv/http/proxmox-pxe/` so every hypervisor netboots the same installer and gets its per-host `answer.toml` from the MAC-matching responder). Must complete before `17-host` runs against the wider `physical` group, since fresh workers need `bootserv01.lan` to netboot AND fresh hypervisors need the published artifacts to consume. Re-imports the 17-host / 20-hypervisor playbooks with `target_hosts: foundation` rather than duplicating them; those playbooks default back to `physical` / `hypervisors` when the var is unset. Driven by `just do-foundation`.
- `17-host/` — Baseline for every baremetal Linux host (`physical` = hypervisors + workers): `10-bootstrap (ansible/pani/tourmanager users) → 15-storage (host-disks role dispatched on host_disks_action: default 'info' reports PRESENT/MISSING per declared partition and never mutates; 'overwrite' is the DESTRUCTIVE LVM-thin carve from inventory storage.disks; 'skip' is silent — no-op on hosts without a storage block) → 16-nfs-mounts (NFS shares declared in nfs_mounts; default empty for workers, hypervisors override to add proxmox RW) → 20-ssh-hardening → 30-firewall → 40-tailscale → 50-node-exporter`. Every playbook's `hosts:` is parameterized as `"{{ target_hosts | default('physical') }}"` so 13-foundation can re-import scoped to a single host; foundation hypervisors see a no-op second pass when this tier runs against the full `physical` group. `just disk-plan` runs 15-storage.yml in info mode.nfs_mounts
- `20-hypervisor/` — Proxmox-only fabric (PVE-specific follow-up after 17-host built the LVM substrate): `10-bootstrap (repo flip) → 20-cluster (gated off, refuses any non-empty pve_cluster) → 40-templates (consume-only: fetches the centrally-baked qcow2 + vzdump tarball from http://bootserv01.lan/ and qm/pct templatize locally; producer is 13-foundation/95-bake-publish-images.yml)`. **PVE lvmthin storage registration (`pvesm add lvmthin vms`) is no longer a step in this tier** — it was folded into the `host-disks` role (`17-host/15-storage.yml`), which auto-publishes `vms` on hypervisors in overwrite mode (see Storage carving). Targets `hypervisors`; same `target_hosts | default('hypervisors')` parameterization as 17-host on 10-bootstrap / 40-templates (20-cluster targets `pve_cluster` and is NOT parameterized — cluster setup is orthogonal to foundation). The old `40-bake-images.yml` (Packer → NAS-publish alternate path) was deleted when the produce-once-on-foundation-publish-to-bootserv flow landed.
- `30-guests/` — OpenTofu barrier (`10-opentofu.yml` + guest-config plays); the single point where the router + hypervisor tracks converge and guests come into existence. `10-opentofu.yml` runs the **un-targeted** `tofu apply` (every VM + LXC) and reconciles the infra LXCs already created by 13-foundation as a no-op. bootserv01 configuration (`15-bootserv.yml`) moved to `13-foundation/90-bootserv.yml` — by the time this tier runs, the bootserv role has already been applied.
- `40-kube/` — guest config: `10-longhorn-host-prep → 20-k3s`

Each playbook self-bootstraps (`../tasks/detect-bootstrap-user.yml`) and asserts its own pre-conditions, so the ordering is the happy path, not the only path — any single tier playbook is runnable in isolation for piecemeal dev (`just run --limit <host>` then fzf-pick it).

**Bootstrap ordering** (the recipes that exist as `just`-shortcuts; the full chain is `just run` + `site.yml`):
1. `just do-router-dhcp` — push dnsmasq static DHCP+DNS leases onto OPNsense for every physical-host MAC
2. `just do-foundation` — bring up (or reconcile) pve-home-01's full 17-host + 20-hypervisor stack, plus the bootserv01 LXC + bootserv role. Run BEFORE any fresh-baremetal install — workers need `bootserv01.lan` to netboot.
3. `just do-host-init` — apply the full 17-host tier to every `physical` host: users (ansible / pani / tourmanager), SSH hardening, firewall, tailscale, node-exporter. Foundation hypervisors see a no-op pass.
4. `just do-bootserv-config` — re-render bootserv01's iPXE + preseed templates (fast iteration during the netboot trial; runs the same playbook 13-foundation does)
5. `just disk-plan` — read-only: run 15-storage.yml with `host_disks_action=info` to report PRESENT/MISSING per declared partition on every physical host (hypervisors + workers). Safe anytime — never halts. Use to confirm a node is compliant with its declared `storage.disks` after a disk add/swap.

The PVE tier (`20-hypervisor/` — cluster join, PVE storage registration, template bake) and 30-guests / 40-kube don't have dedicated `just-do-X` recipes; run them via `just run` (fzf-picks a playbook) or by invoking the tier's `site.yml` directly. The LVM carve (gated by `host_disks_action: overwrite`) lives in `17-host/15-storage.yml`; `just do-host-init` runs the full 17-host tier with 15-storage in info mode by default, so already-up hosts do not halt — set `host_disks_action: overwrite` per-host or per-run to actually carve.

Full end-to-end is `playbooks/poochella/site.yml` (imports `infra/site.yml`); `infra/site.yml` imports the seven tier `site.yml`s in order, and each tier `site.yml` imports its playbooks in ordinal order.

### Vault

- Indirection layer: roles/playbooks reference unprefixed names (`advanceteam_user_pass`, `k3s_cluster_token`, …) defined in `group_vars/all/vars.yml`, which re-export `vault_*` values from the encrypted `group_vars/all/vault.yml`. When adding a new secret, edit both files.
- **Proxmox Tofu auth (shared service account)**: Tofu authenticates to EVERY standalone hypervisor as one dedicated PVE-realm user, `advanceteam@pve`, granted a scoped custom `Terraform` ACL role on `/` (the bpg/proxmox provider's recommended privilege set). The account is created identically on each node by `20-hypervisor/15-tofu-service-account.yml` (password set on create from the vault; idempotent). ONE well-known password authenticates all nodes — `advanceteam_user_pass` (re-export of `vault_advanceteam_user_pass`), passed to each tofu workspace (in the single `tofu/node/` root) via `TF_VAR_proxmox_password`; the username is the Tofu var `proxmox_username` (default `advanceteam@pve`). Adding a hypervisor needs **no new secret** — the account already exists once the 20-hypervisor tier has run against it. Rotate by `pveum passwd advanceteam@pve` on every node + re-encrypting `vault_advanceteam_user_pass`. The legacy per-node `root@pam!tofu-lan` API tokens (old `proxmox_api_tokens` map) were retired in this cutover; the tokens + their `vault_proxmox_api_token_pve_home_*` keys still exist as an unused fallback.
- **Break-glass account**: the `tourmanager` user is the ONLY account that can SSH with a password after `17-host/20-ssh-hardening.yml` runs (a `Match User tourmanager` block in the sshd drop-in keeps password auth for it alone). The password lives in `vault_tourmanager_user_pass`.

## Architecture

### Inventory (`inventory.yaml`)

Single source of truth for every host in the homelab. Read by Ansible (as its inventory; see `ansible.cfg`), by OpenTofu via `yamldecode` (`tofu/modules/hypervisor/locals.tf` + each `tofu/per-node/<host>/main.tf`), and by the dnsmasq playbook (which walks hostvars to build static reservations).

Per-host fields:
- `ansible_host` — DNS name or IP Ansible uses to reach the host
- `ip_address` + (`mac_addresses` for physical hosts | `mac_address` for VMs) — host appears as a dnsmasq static DHCP+DNS reservation on OPNsense. Physical hosts use a LIST of every NIC's MAC (multi-NIC hosts collapse to one OPNsense entry with all MACs in `hardware_addr`, and every entry is also a valid iPXE boot MAC). VMs have exactly one NIC and so a single `mac_address` string, which is also the MAC Tofu provisions on the VM NIC.
- `static_ip: true` — suppress the dnsmasq reservation (e.g. the router itself)
- `vm: { proxmox_node, cores, memory, disk_size, tags, ... }` — Tofu creates this VM on Proxmox, pinned to `proxmox_node`. VM MACs use the locally-administered `02:` prefix. VM boot disks live on the node-local `vms` PVE storage; no VM ever receives a Longhorn-backed data disk (replicas live only on baremetal kube_workers — see Storage carving below).

Groups define topology:
- `router` → OPNsense (`opnsense01` at `10.1.1.1`)
- `physical` → umbrella for **every** baremetal Linux box (the 17-host tier targets this). Children:
  - `hypervisors` → PVE hosts only (the 20-hypervisor tier targets this). Children:
    - `pve_standalone` → every PVE host lives here (today: `pve-home-01` only). No corosync, no shared storage.
    - `pve_cluster` → **gated off**: empty group kept as deprecated scaffolding only. `20-hypervisor/20-cluster.yml` asserts this group is empty and fails the play otherwise.
  - `workers` → plain Linux baremetal (no PVE). k3s baremetal workers, app hosts, etc. Currently: `worker-home-02` (also listed in `kube_workers`). Workers have no `vm:`/`lxc:` block and are not in the `hypervisors` group, so Tofu does not see them.
- `foundation` → top-level group containing the hypervisor(s) that must be all-the-way-up (full stack + bootserv01 LXC + bootserv role) before any other baremetal can come into being. Targeted by the 13-foundation tier via `target_hosts: foundation`. Today: just `pve-home-01`. Members MUST also appear under `hypervisors`.
- `switches` → managed network gear that wants a static lease
- `virtual-machines` → has children `databases` and `kubernetes`
- `kubernetes` → children `kube_control_plane` (kube-ctl-01..03) and `kube_workers` (kube-worker-01; -02/-03 commented out). All VM-based kube nodes are pinned to `pve-home-01` while it is the only hypervisor.
- `containers` → `bootserv` (bootserv01 LXC on pve-home-01) + `webservers` placeholder

Each hypervisor's API endpoint is **derived by convention** from its hostname — `https://<hypervisor_name>.<domain>:8006` (built in `tofu/node/main.tf` from `hypervisor_name` + `all.vars.domain`); there is no endpoint map. `var.proxmox_endpoint` remains an optional override for manual CLI use (an IP or alt port). Adding a hypervisor: see `tofu/README.md` — just add the inventory host entry; no endpoint to declare, no per-node template ID (fleet-wide `template_*_id_base` scalars), no directory to copy and no per-node secret (the shared `advanceteam@pve` account authenticates everywhere, and the Ansible drivers create a fresh tofu workspace for the node on first apply).

### Provisioning flow

VMs are declared once in `inventory.yaml` (any host with a `vm:` block). A SINGLE flat Tofu root (`tofu/node/`) serves every hypervisor; each node gets its own state via a named tofu **workspace** (`tofu/node/terraform.tfstate.d/<host>/`), selected per apply along with the node's `hypervisor_name` (from which the API URL is derived). The root calls `tofu/modules/hypervisor`, which walks `inventory.yaml`, slices it to guests pinned to that hypervisor, and instantiates the shared `tofu/modules/vm` + `tofu/modules/lxc` modules. The MAC declared in inventory is set on the VM NIC, so the static DHCP reservation pushed to OPNsense in step 1 of the bootstrap (`just do-router-dhcp`) lights up the moment the VM DHCPs.

**State-per-hypervisor**: standalone hypervisors don't share an API and don't share Tofu state. Isolation is by named tofu **workspace** (one per host) inside the single flat root, which has one non-aliased `proxmox` provider whose endpoint is derived from `hypervisor_name` (`https://<host>.<domain>:8006`) — adding a hypervisor is an inventory edit only, no directory copy (see `tofu/README.md`). No alias gymnastics, no cross-node coupling at the Terraform layer.

`playbooks/poochella/infra/30-guests/10-opentofu.yml` loops over the `hypervisors` group and per-iteration:
1. Asserts the shared service-account password (`advanceteam_user_pass`) is set in the vault **and** that the `advanceteam@pve` user exists on that node via its own `pveum user list`
2. Runs `community.general.terraform` against the flat `tofu/node/` root with `workspace: <that hypervisor>` + the `hypervisor_name` var (Tofu derives the endpoint from it), passing the one shared password via `TF_VAR_proxmox_password` (sourced from `advanceteam_user_pass`); the Tofu `proxmox_username` var defaults to `advanceteam@pve`.

`playbooks/poochella/infra/13-foundation/80-tofu-infra-lxcs.yml` loops over the `foundation` group with the same shape, but with `targets: [module.hypervisor.module.infra_lxcs]` on the `community.general.terraform` call — so it creates bootserv01 without incurring the cost (or risk) of materializing the VMs in that hypervisor's tofu workspace. Targeting is per-invocation against that node's workspace state; the later un-targeted apply reconciles everything (the LXC ends up as a no-op).

VM disks live on the **node-local** `vms` LVM-thin storage (same storage ID on every hypervisor, each backed by that node's own VG). Templates are **produced once on foundation, published to bootserv01, consumed by every hypervisor**: `13-foundation/95-bake-publish-images.yml` runs on the foundation node — downloads the pinned upstream Debian cloud image, `virt-customize`s + `virt-sysprep`s the qcow2, snapshots the local LXC template (built by `50-bootstrap-lxc-template.yml`) via `vzdump`, then `pct push`es both artifacts + sha256 sidecars into bootserv01's `/srv/http/cloud-images/` and `/srv/http/lxc-templates/`. Every Proxmox node (including foundation) then runs `20-hypervisor/40-templates.yml` which fetches from `http://bootserv01.lan/...`, verifies sha256, and `qm`/`pct template`s locally — peers get a byte-identical copy of foundation's template. Foundation's LXC `pct status` check short-circuits the LXC consume step (the local template from the bootstrap pass satisfies it); foundation does a self-fetch loopback for the VM template (cheap, keeps the consume code path symmetric). The shared image manifest (`vm_templates_base` / `lxc_templates_base`) lives in `playbooks/poochella/infra/vars/templates.yml`, loaded by both the producer and the consumer playbooks. Cloud image URLs are **pinned to dated upstream snapshots** — re-pin by bumping `image_url` + `image_file` together; producer sees no match on bootserv01 and rebakes+republishes; consumers see no local VMID/CTID match for the new artifact and re-templatize. The template VMID/CTID comes from the fleet-wide `template_vm_id_base` / `template_ct_id_base` scalars in `inventory.yaml` (single source of truth, also read by `tofu/modules/hypervisor/locals.tf`). Standalone hypervisors each own an independent VMID namespace (no shared cluster config DB), so the template anchor is the SAME ID on every node — not a per-host map; `qm/pct clone <id>` on each node clones that node's own local template. Real guests are auto-assigned from 100 by PVE per-node (Tofu sets no `vm_id` on the guest, only `clone.vm_id`). Each VM is created on and pinned to its `proxmox_node` and cloned from that node's local template. The bpg/proxmox v0.106 `initialization` block has no `hostname` field — VM `name` propagates to cloud-init instead. DNS + search domain reach guests via DHCP options set on OPNsense (single source of truth for DNS).

### Storage carving (DESTRUCTIVE)

**Declared, not discovered — but by stable selector, not device name.** Each physical host's disk layout is the `storage.disks` list in `inventory.yaml` (single source of truth; the schema is the SAME for hypervisors and workers — the role distinguishes them by which fields each partition declares). Each disk has a `select:` (`boot` | `{model,serial}` | `{min_size_gib|size_gib, rotational}`), an optional per-disk `wipe: refuse|force`, and an ordered `partitions:` list. In OVERWRITE mode a non-`boot` selector **must resolve to exactly one** non-boot, non-removable disk — 0 or >1 matches is a **hard preflight failure** (the role never guesses, never auto-claims). In INFO mode the same zero/many path is tolerant and emits `MISSING (disk not found)` for the orphan partitions instead of halting. Selectors match hardware attributes, so kernel renames (`sdX`/`nvmeXnY` shuffle on disk add) are irrelevant and swapping a disk for an identical/larger one needs **no inventory edit** when the selector is attribute-based. `just disk-plan` (read-only `15-storage.yml` with `host_disks_action=info`) reports PRESENT/MISSING per declared partition on every physical host. The paste-ready selector skeleton previously emitted by `disk-plan` is still on disk as `roles/host-disks/tasks/discover.yml` but is no longer wired into any `just` recipe.

Each partition declares `label` (GPT partlabel + idempotency key, **unique per node**), `vg`, `thinpool`, `size` (sgdisk `+SIZE`; `0` = fill), then **EXACTLY ONE** of (XOR, enforced in preflight): `pve_storage` (a **frozen contract**: `tofu/modules/vm` hardcodes `vms`; Tofu does *not* read `storage:` so the schema is Ansible-only) + `content`, OR `mount: { path, fstype, lv_name?, lv_size? }` (the role builds a thin LV, ext4s it, and fstab-mounts it). Several partitions on **different physical disks** may share one `vg`/`thinpool` — that is how a NVMe + HDD present as one larger pool (e.g. `longhornthin` mounted at `/var/lib/longhorn` on a baremetal `kube_worker`; the `mount:` block lives on exactly ONE partition per pool). The hypervisor-only convention (asserted in preflight, see `group_vars/hypervisors.yml`): every hypervisor yields `vms` (universal — every VM/LXC boot disk + cloud-init drive uses it). That is the only hypervisor-side PVE storage in the contract — Longhorn replica state lives on baremetal kube_workers only, never on a hypervisor pool, so a small boot NVMe is sufficient on every hypervisor (pve-home-01 today: 256 GB NVMe, vms only).

`17-host/15-storage.yml` invokes the `host-disks` role, which dispatches via `tasks/main.yml` on `host_disks_action`: `info` runs `preflight.yml` + a read-only `inspect.yml` that never halts on missing partlabels or missing disks; `overwrite` runs `preflight.yml` → `carve.yml` (→ `_wipe_disk.yml` + `_carve_one.yml`) → `storage_pools.yml` (the destructive path — explicit consent); `skip` is a single `meta: end_host`. Preflight resolves every selector to a device and (overwrite mode only) asserts the partlabel-uniqueness and pool-target-consistency invariants. Hypervisor-only preflight steps (gated on overwrite mode AND on `inventory_hostname in groups['hypervisors']`): the `qm/pct list` + template detection guards, plus the frozen-contract assertion that this hypervisor yields `vms`. Overwrite mode is "nuke from orbit": preflight stops every running VM/CT (`qm/pct stop`, 60s graceful) then `qm/pct destroy --purge`es every guest on this hypervisor — real and template — so the carve can rebuild the storage they sat on. Real guests come back via the un-targeted `tofu apply` in `30-guests/10-opentofu.yml`; regenerable templates come back via `20-hypervisor/40-templates.yml` (consume from bootserv01). If the foundation node itself is being carved, that pass also wipes the bootstrap LXC template — the next `13-foundation/site.yml` run rebakes it via `50-bootstrap-lxc-template.yml` and re-publishes via `95-bake-publish-images.yml`. The legacy `carve_destroy_templates` flag and the real-guest hard refusal are both gone — `host_disks_action: overwrite` IS the consent. Carve runs a per-disk **wipe gate** once per disk (boot disk never wiped; a disk already carrying our partlabels is skipped — idempotent; a disk with **foreign signatures** is REFUSED unless it declares `wipe: force`), then carves each partition in declared order (idempotent on the GPT partlabel). `storage_pools.yml` builds the `vg`/`thinpool` LVM-thin pool (a VG may span multiple PVs), **grows** an existing thin pool onto a newly-fitted larger disk (`lvextend +100%FREE`), and — for any partition with `mount:` — creates the thin LV, ext4-formats it, and persists the mount in fstab. The PVE-side publish step is folded into the same role (`tasks/pve_register.yml`, ordered after the pool build): on hypervisors it runs `pvesm add lvmthin` for each unique `pve_storage`, scoped `--nodes <self>` (independent nodes ⇒ per-node `/etc/pve/storage.cfg`) — automatically in overwrite mode, and in info mode only when the caller sets `host_disks_register_pve: true` (the foundation tier does, to publish on its read-only reconcile without re-carving; `just disk-plan` omits it and stays read-only). Disk swap / capacity / NAS-backup runbook: `docs/storage-disk-runbook.md`. **Data safety is a Longhorn-layer property (separate Flux repo), not an LVM one** — the carve only makes the empty re-init safe; Longhorn's replica-2 + zone anti-affinity is what survives a disk pull.

Longhorn placement: replica state lives ONLY on baremetal `kube_workers` (today: `worker-home-02`). Each baremetal worker declares `storage.disks` with a `mount: /var/lib/longhorn ext4` partition that `host-disks` realises directly from `17-host/15-storage.yml` — no VM ever receives a Longhorn-backed data disk, and no hypervisor pool backs Longhorn. `40-kube/10-longhorn-host-prep.yml` runs cluster-wide and installs ONLY the consumer prereqs (open-iscsi + nfs-common + iscsid) so the schedulable VM control planes can attach + mount Longhorn volumes over iSCSI/NFS. The Flux-managed Longhorn install must restrict storage scheduling to the workers (`createDefaultDiskLabeledNodes` + the `node.longhorn.io/create-default-disk=true` label set on `kube_workers` only) so replica + rebuild IO stays on the baremetal workers and never lands on the etcd CP VMs.

### k3s + kube-vip

`playbooks/poochella/infra/40-kube/20-k3s.yml` builds an embedded-etcd HA cluster (k3s runs inside VM-based control planes, plus an agent on the baremetal `worker-home-02`; CP VMs are all pinned to `pve-home-01` while it is the only hypervisor, so that host is the etcd blast radius):
1. First CP runs `k3s server --cluster-init`; kube-vip is then deployed only on that node to bring up the L2 VIP at `k3s_kube_vip_address` (currently `10.1.1.50`)
2. Remaining CPs join `serial: 1` via `https://<api-endpoint>:6443` where api-endpoint is `k3s_api_dns_name` if set, else the VIP; workers join after
3. kubeconfig is fetched back to `{{ playbook_dir }}/../../../kubeconfig` (repo root) with its `server:` URL rewritten to the DNS name (else the VIP) — `export KUBECONFIG=$(pwd)/kubeconfig` to use it

The VIP must be a free address on the same L2 segment as the control planes. Config lives in `group_vars/kubernetes.yml`. Layered on top of the VIP is `k3s_api_dns_name: api.doghouse.lan` — pushed to OPNsense Unbound by `10-router/30-unbound.yml` (the more-specific override coexists with the `*.doghouse.lan` ingress wildcard). The k3s role uses the DNS name for the joiner `--server` URL, the rewritten kubeconfig `server:`, and as a TLS SAN on the apiserver cert; kube-vip still owns sub-second L2 failover between CPs, the DNS layer is sugar on top. Renumbering the VIP is therefore a one-line DNS edit. The bootstrap order (10-router before 40-kube) already satisfies the soft DNS dependency the layer introduces.

### Flux / GitOps (`k8s/`) — public/private split

`k8s/` is the cluster's Flux GitOps root and is **PUBLIC, infrastructure-only**. It is the visibility boundary: the platform layer is committed here; **application manifests are private IP and never enter this repo's history** — they live in the separate private repo `github.com/Ivanipani/doghouse`, imported as an **opaque cross-repo source**.

- **Layout** (`k8s/clusters/doghouse/` is the Flux entrypoint): `flux-system/` (gotk-components + gotk-sync pointing at THIS repo, sync path `./k8s/clusters/doghouse`); `infra.yaml` (the infra Kustomizations); `apps.yaml` (the apps Kustomization + a dormant `doghouse-apps` GitRepository). Platform manifests are under `k8s/infra/{networking,storage,monitoring}/{controllers,configs}`. **Flux `spec.path` is repo-root-relative**, so every Kustomization path is prefixed `./k8s/…` — keep that prefix on any new one.
- **Apps import** (`apps.yaml`): today apps still live locally at `k8s/apps/doghouse` and the `apps` Kustomization sources `flux-system` (this repo), so the cluster works. The `doghouse-apps` GitRepository (→ private repo, authed by the read-only `doghouse-apps-key` Secret) is wired but dormant. **One-step cutover later**: move `k8s/apps` into the private repo, then flip the `apps` Kustomization's `sourceRef` `flux-system → doghouse-apps` and `path` `./k8s/apps/doghouse → ./apps/doghouse` (procedure documented at the top of `apps.yaml`).
- **Bootstrap is Ansible-driven**: `40-kube/40-flux.yml` (run via `just do-flux`, host `localhost`) runs `flux bootstrap git` against this repo over **SSH** (GitHub rejects PAT-over-HTTPS for git ops; key staged from `vault_github_ssh_private_key`, `--path=k8s/clusters/doghouse`) using the kubeconfig `20-k3s.yml` fetched, then installs the `sops-age` Secret from `vault_sops_age_key` and the `doghouse-apps-key` Secret from `vault_doghouse_apps_deploy_key` (each skipped if its vault value is unset/placeholder). Idempotent; it commits + pushes `flux-system/` to the repo.
- **SOPS**: encrypted `*.sops.yaml` are committed in the public repo (safe — only the age **public** key is exposed in `k8s/.sops.yaml`); the age **private** key is vaulted as `vault_sops_age_key` and never a loose tracked file (`*.agekey` is git-ignored). Decryption is wired via `spec.decryption` on the `storage-controllers` / `monitoring-controllers` Kustomizations.

### Roles

Roles under `roles/` are units of work, not "things to install":
- `host-base` — every physical-host baseline: install sudo+zsh, import `apt-no-auto-upgrades`, create `ansible`/`pani`/`tourmanager` users via `create-user`, set root password from vault, and deploy the baseline zsh config (`templates/zshrc.local.j2` → `/etc/zsh/zshrc.local`, sourced via a one-line hook appended to the package's `/etc/zsh/zshrc`). zsh is the project-wide default login shell for every baseline user (`image_baseline_users` shells are `/usr/bin/zsh`); the zsh config + shells are also baked into images by `image-baseline.sh.j2` (config body duplicated byte-for-byte to keep the reconcile no-op), so `zsh` is in every medium's pre-baseline package set (`vm_packages` / `lxc_packages` / preseed `pkgsel/include`). Driven by `17-host/10-bootstrap.yml`. The user policy is in `defaults/main.yml`'s `host_base_users`; the zsh contract is `image_baseline_zsh` in `group_vars/all/image-baseline.yml`.

Its `cluster_ssh.yml` / `cluster_create.yml` / `cluster_join.yml` task files are **gated off** by an assertion in `20-hypervisor/20-cluster.yml` and kept on disk as deprecated reference. - `host-disks` — **declared, selector-based** (inventory `storage.disks`) disk carving + node-local LVM-thin pool provisioning for every physical host (hypervisors + workers). Each partition declares EITHER `pve_storage` + `content` (published as PVE lvmthin by the role's own `tasks/pve_register.yml`) OR `mount: { path, fstype, ... }` (role builds a thin LV, ext4s it, fstab-mounts it — used for `/var/lib/longhorn` on baremetal kube_workers). ## Conventions

- Playbook filenames are prefixed `NN-` to encode run order within `infra/`. Don't reorder by renumbering unless you also understand the dependencies.
- The Ansible remote user is `ansible` with key `~/.ssh/ansible` (set in `ansible.cfg`). Cloud-init bakes this user into VM templates (the producer step `13-foundation/95-bake-publish-images.yml`); the `host-base` role creates it on every physical host.
- Pre-conditions are documented as comment blocks at the top of each infra playbook — read those before editing.
- `playbooks/wip/` is scratchpad / in-progress work, not part of `site.yml`.

## Collaboration rules (for Claude)

- **Diagnose by running read-only ansible ad-hoc commands directly**, e.g. `ansible pve-home-01 -m shell -a 'pveam list local' --become`, `ansible bootserv01 -m shell -a 'systemctl is-active dnsmasq nginx' --become`. Do not ask the user to copy/paste terminal output when an ad-hoc command can fetch the same information. The `ansible` user + `~/.ssh/ansible` key already trusts every fleet host.
- **Never run a destructive command on the fleet without explicit per-command user permission.** "Destructive" = anything that mutates state on the host (`pct destroy`, `pveam remove`, `pvesm remove`, `rm`, `systemctl stop/disable`, `apt remove`, `qm destroy`, partitioning / disk wipes, killing processes, writing files outside `/tmp`, etc.). Reading config, listing resources, dumping logs, and `--check`/`--diff` dry-runs are fine. The same rule applies to OpenTofu (`tofu apply` against existing resources is destructive in principle) and to anything that touches OPNsense state.
- **Use the NATO alphabet when naming cluster entities. Always start at Alpha move up.**

## bootserv01 / iPXE boot flow

`bootserv01` is an LXC on `pve-home-01` (inventory `containers.bootserv.bootserv01`, `lxc:` block tagged `infra`) that serves TFTP (iPXE chainload binaries) and HTTP (boot scripts + Debian netboot kernel/initrd + per-host preseeds + the Proxmox VE auto-install artifacts) to baremetal hosts that net-boot from their NIC. OPNsense's dnsmasq advertises `ipxe.efi` / `undionly.kpxe` to vanilla-PXE clients (chainload) and a script URL to user-class iPXE clients. Per-host iPXE dispatch is rendered by the `bootserv` Ansible role from every inventory host whose `storage.disks` carries a `select: boot` entry with an `hw: {model, serial}` pin (the marker that the host is installable via this bootserv). The dispatcher matches against every entry in that host's `mac_addresses` list — there is no separate "boot MAC" field.

**Two install classes, branched by group.** `boot.ipxe` routes a host's MACs to the **Debian preseed** path (`hosts/<host>.ipxe` → d-i + per-host preseed) when the host is NOT a hypervisor, or to the **Proxmox VE auto-installer** when it IS in `groups['hypervisors']` (chains the single shared `proxmox-install.ipxe` — there is no per-host PVE iPXE file). PVE uses **ONE shared installer** for the whole fleet; per-host selection happens at install time via the **MAC-matching answer responder**. The producer `13-foundation/97-bake-publish-pve-pxe.yml` downloads the pinned PVE ISO once on foundation (`pve_pxe` in `vars/templates.yml`), **patches the installer's dhclient `timeout 10;`→`60;` inside `pve-installer.squashfs`** (extract squashfs → `unsquashfs` → `sed` → `mksquashfs` → rebuild a bootable ISO with xorriso's own reported El-Torito/GPT options so partitions recompute for the bigger squashfs), then runs `proxmox-auto-install-assistant prepare-iso --fetch-from http --url http://bootserv01.lan/proxmox-answer --pxe-loader ipxe` and `pct push`es `vmlinuz` + `initrd.img` + the stripped iso flat to `/srv/http/proxmox-pxe/` (+ sha256 sidecars). The heavy bake runs **detached** (`files/bake-pve-pxe.sh.j2`, `setsid`+`nice`, sentinel-polled) because it saturates the mini-PC and drops a held SSH channel. **Why patch + DHCP + shared ISO, not static + per-host baked:** the stock PVE installer fires one short ~10 s DHCP burst (`dhclient -v` from `unconfigured.sh`) before the NIC link finishes re-negotiating after the kernel driver reset — no tunable timeout — so `from-dhcp` races and fails on slow-link NICs (iPXE works only because it never drops the link). This is **Proxmox bug #5583**; the fix is to raise the dhclient `timeout` so it spans the link-up window. (A static-network `from-answer` ISO baked per-host was tried first and abandoned: it still failed DNS when first-boot fetched scripts, and it meant maintaining an ISO per host.) The booting node POSTs its sysinfo to `/proxmox-answer`; the responder returns that host's `answer.toml` (`[network] source = "from-dhcp"`, NIC pinned by MAC, disk by `[disk-setup]` serial filter). First-boot runs the **canonical `image-baseline.sh`** via `[first-boot] source = "from-url"` → the LAN-only `/scripts/image-baseline.sh`. The `img/proxmox/` ISO path (used to bootstrap the foundation node itself, before bootserv01 exists) embeds the same baseline via `--on-first-boot`. Netbootable hypervisor = a `hypervisors` member with a `select: boot` disk carrying an `hw: {model, serial}` pin.

**Brought up by the 13-foundation tier.** The bootserv01 LXC is created by `13-foundation/80-tofu-infra-lxcs.yml` (loops over the `foundation` group and runs `tofu apply -target=module.hypervisor.module.infra_lxcs` against the flat `tofu/node/` root in each foundation node's tofu workspace — does NOT touch any VM) and configured by `13-foundation/90-bootserv.yml` (was `30-guests/15-bootserv.yml`). The PVE PXE artifacts are then published by `13-foundation/97-bake-publish-pve-pxe.yml` (after the VM/LXC producer `95-` and the mirror `96-`). Foundation must complete before any other baremetal can be netbooted — that's the whole point of the tier.

**Tofu ordering invariant (still relevant for the un-targeted per-node apply in 30-guests)**: LXCs whose `lxc.tags` contains `infra` come up before any VMs. In `tofu/modules/hypervisor/main.tf`, `module.vms` declares `depends_on = [module.infra_lxcs]`, so even when the un-targeted apply runs first on a fresh fleet (no foundation tier run beforehand), bootserv01 still comes up before kube/dev VMs on the same hypervisor.
