# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Declarative homelab automation for **poochella**: a small Proxmox VE fleet running a k3s Kubernetes cluster on node-local LVM-thin-backed VMs, fronted by an OPNsense router. OpenTofu provisions infrastructure on Proxmox; Ansible configures everything else. The control node is the user's laptop.

**Physical hosts split by role, not by hardware.** Today: `pve-home-01` is the only Proxmox hypervisor (mini PC); `worker-home-02` (renamed from `pve-home-02`) is plain Debian 13 baremetal (gaming-PC hardware that replaced the old crash-looping unit, since de-Proxmoxed for use as a Linux worker ‚Äî both VM-host and baremetal kube_worker). `pve-home-03` was retired. Inventory umbrella groups: `physical` = every baremetal Linux box (`hypervisors` ‚à™ `workers`); `hypervisors` is PVE-only; `workers` is plain Linux.

**Every Proxmox hypervisor is a fully standalone PVE host** ‚Äî **no corosync, no shared storage, no HA, no live-migration**. The cluster code path is **gated off**: an assertion at the top of `20-hypervisor/20-cluster.yml` refuses any non-empty `pve_cluster` group, and the cluster task files under `roles/hypervisor/` are kept on disk only as deprecated reference.


## Common commands

All workflows are wrapped in `justfile` recipes ‚Äî prefer `just <recipe>` over invoking `ansible-playbook` directly:

- `just check` ‚Äî verify control-node deps (ansible, fzf, collections, `passlib`)
- `just install` ‚Äî install ansible-core via `uv tool install` plus required collections
- `just ping` ‚Äî `ansible all -m ping` smoke test
- `just run [options]` ‚Äî fzf-pick a playbook under `playbooks/` and run it (auto-passes `--vault-password-file ansible-pass`)
- `just test` ‚Äî fzf-pick tags from `test/test.yml` and run that subset
- `just ssh-refresh` ‚Äî clear & re-seed `known_hosts` for every host in the inventory (useful after re-imaging)
- `just secret-encrypt <name>` ‚Äî `ansible-vault encrypt_string` for adding values to `group_vars/all/vault.yml`
- Tofu is driven by Ansible (`13-foundation/80-tofu-infra-lxcs.yml` + `30-guests/10-opentofu.yml`); for manual per-hypervisor work, `cd tofu/per-node/<host>` and run `tofu init/plan/apply` directly (see `tofu/README.md`)

**Infra layout (tier directories).** `playbooks/poochella/infra/` is **seven** ordered tiers, each a directory with its own ordered `site.yml`; ordinals are gap-10 *within* a tier so steps insert without renumbering. The directory **is** the target tier:
- `10-router/` ‚Äî OPNsense DNS/DHCP foundation (precedes everything)
- `15-nas/` ‚Äî Synology NAS (gap-5 break; independent of host bootstrap, admin'd via DSM rather than the ansible-user path)
- `13-foundation/` ‚Äî Foundational subset (the `foundation` inventory group, today just `pve-home-01`): full 17-host + 20-hypervisor stack against that host, then **bootstrap-only LOCAL LXC template bake** (`50-bootstrap-lxc-template.yml` ‚Äî breaks the chicken-and-egg: bootserv01 LXC needs a Debian LXC template to exist on the foundation node before Tofu can clone it), then a **TARGETED** `tofu apply -target=module.infra_lxcs_<node>` that creates ONLY the infra LXCs (bootserv01 today) ‚Äî never any VM ‚Äî then the bootserv role, then the **full PRODUCER step** (`95-bake-publish-images.yml`: bake the Debian VM cloud image via `virt-customize`+`virt-sysprep` and snapshot the local LXC template via `vzdump`, then PUBLISH both to bootserv01 via `pct push` at `/srv/http/cloud-images/<image_file>` + `/srv/http/lxc-templates/<artifact>` with sha256 sidecars). Must complete before `17-host` runs against the wider `physical` group, since fresh workers need `bootserv01.lan` to netboot AND fresh hypervisors need the published artifacts to consume. Re-imports the 17-host / 20-hypervisor playbooks with `target_hosts: foundation` rather than duplicating them; those playbooks default back to `physical` / `hypervisors` when the var is unset. Driven by `just do-foundation`.
- `17-host/` ‚Äî Baseline for every baremetal Linux host (`physical` = hypervisors + workers): `10-bootstrap (ansible/pani/tourmanager users) ‚Üí 15-storage (host-disks role dispatched on host_disks_action: default 'info' reports PRESENT/MISSING per declared partition and never mutates; 'overwrite' is the DESTRUCTIVE LVM-thin carve from inventory storage.disks; 'skip' is silent ‚Äî no-op on hosts without a storage block) ‚Üí 16-nfs-mounts (NFS shares declared in nfs_mounts; default empty for workers, hypervisors override to add proxmox RW) ‚Üí 20-ssh-hardening ‚Üí 30-firewall ‚Üí 40-tailscale ‚Üí 50-node-exporter`. Every playbook's `hosts:` is parameterized as `"{{ target_hosts | default('physical') }}"` so 13-foundation can re-import scoped to a single host; foundation hypervisors see a no-op second pass when this tier runs against the full `physical` group. `just disk-plan` runs 15-storage.yml in info mode.nfs_mounts
- `20-hypervisor/` ‚Äî Proxmox-only fabric (PVE-specific follow-up after 17-host built the LVM substrate): `10-bootstrap (repo flip) ‚Üí 20-cluster (gated off, refuses any non-empty pve_cluster) ‚Üí 30-pve-storage-register (publishes the LVM-thin pool from 17-host as the PVE lvmthin storage under the frozen-contract name vms) ‚Üí 40-templates (consume-only: fetches the centrally-baked qcow2 + vzdump tarball from http://bootserv01.lan/ and qm/pct templatize locally; producer is 13-foundation/95-bake-publish-images.yml)`. Targets `hypervisors`; same `target_hosts | default('hypervisors')` parameterization as 17-host on 10-bootstrap / 30-pve-storage-register / 40-templates (20-cluster targets `pve_cluster` and is NOT parameterized ‚Äî cluster setup is orthogonal to foundation). The old `40-bake-images.yml` (Packer ‚Üí NAS-publish alternate path) was deleted when the produce-once-on-foundation-publish-to-bootserv flow landed.
- `30-guests/` ‚Äî OpenTofu barrier (`10-opentofu.yml` + guest-config plays); the single point where the router + hypervisor tracks converge and guests come into existence. `10-opentofu.yml` runs the **un-targeted** `tofu apply` (every VM + LXC) and reconciles the infra LXCs already created by 13-foundation as a no-op. bootserv01 configuration (`15-bootserv.yml`) moved to `13-foundation/90-bootserv.yml` ‚Äî by the time this tier runs, the bootserv role has already been applied.
- `40-kube/` ‚Äî guest config: `10-longhorn-host-prep ‚Üí 20-k3s`

Each playbook self-bootstraps (`../tasks/detect-bootstrap-user.yml`) and asserts its own pre-conditions, so the ordering is the happy path, not the only path ‚Äî any single tier playbook is runnable in isolation for piecemeal dev (`just run --limit <host>` then fzf-pick it).

**Bootstrap ordering** (the recipes that exist as `just`-shortcuts; the full chain is `just run` + `site.yml`):
1. `just do-router-dhcp` ‚Äî push dnsmasq static DHCP+DNS leases onto OPNsense for every physical-host MAC
2. `just do-foundation` ‚Äî bring up (or reconcile) pve-home-01's full 17-host + 20-hypervisor stack, plus the bootserv01 LXC + bootserv role. Run BEFORE any fresh-baremetal install ‚Äî workers need `bootserv01.lan` to netboot.
3. `just do-host-init` ‚Äî apply the full 17-host tier to every `physical` host: users (ansible / pani / tourmanager), SSH hardening, firewall, tailscale, node-exporter. Foundation hypervisors see a no-op pass.
4. `just do-bootserv-config` ‚Äî re-render bootserv01's iPXE + preseed templates (fast iteration during the netboot trial; runs the same playbook 13-foundation does)
5. `just disk-plan` ‚Äî read-only: run 15-storage.yml with `host_disks_action=info` to report PRESENT/MISSING per declared partition on every physical host (hypervisors + workers). Safe anytime ‚Äî never halts. Use to confirm a node is compliant with its declared `storage.disks` after a disk add/swap.

The PVE tier (`20-hypervisor/` ‚Äî cluster join, PVE storage registration, template bake) and 30-guests / 40-kube don't have dedicated `just-do-X` recipes; run them via `just run` (fzf-picks a playbook) or by invoking the tier's `site.yml` directly. The LVM carve (gated by `host_disks_action: overwrite`) lives in `17-host/15-storage.yml`; `just do-host-init` runs the full 17-host tier with 15-storage in info mode by default, so already-up hosts do not halt ‚Äî set `host_disks_action: overwrite` per-host or per-run to actually carve.

Full end-to-end is `playbooks/poochella/site.yml` (imports `infra/site.yml`); `infra/site.yml` imports the seven tier `site.yml`s in order, and each tier `site.yml` imports its playbooks in ordinal order.

### Vault

- Indirection layer: roles/playbooks reference unprefixed names (`proxmox_api_token`, `k3s_cluster_token`, ‚Ä¶) defined in `group_vars/all/vars.yml`, which re-export `vault_*` values from the encrypted `group_vars/all/vault.yml`. When adding a new secret, edit both files.
- **Per-hypervisor Proxmox tokens**: standalone nodes don't share `/etc/pve`, so each PVE host has its own `tofu-lan` API token. `group_vars/all/vars.yml` exposes a `proxmox_api_tokens` map keyed by hypervisor name (workers don't run the PVE API and don't appear here). Add a hypervisor = mint with `pveum user token add root@pam tofu-lan --privsep 0` on that node, then `just secret-encrypt vault_proxmox_api_token_<host>`. See the header of `30-guests/10-opentofu.yml`. Each iteration of the Ansible loop passes ONE token via `TF_VAR_proxmox_api_token` to the matching `tofu/per-node/<host>/` workspace, which exposes a single `var.proxmox_api_token` string.
- **Break-glass account**: the `tourmanager` user is the ONLY account that can SSH with a password after `17-host/20-ssh-hardening.yml` runs (a `Match User tourmanager` block in the sshd drop-in keeps password auth for it alone). The password lives in `vault_tourmanager_user_pass`.

## Architecture

### Inventory (`inventory.yaml`)

Single source of truth for every host in the homelab. Read by Ansible (as its inventory; see `ansible.cfg`), by OpenTofu via `yamldecode` (`tofu/modules/hypervisor/locals.tf` + each `tofu/per-node/<host>/main.tf`), and by the dnsmasq playbook (which walks hostvars to build static reservations).

Per-host fields:
- `ansible_host` ‚Äî DNS name or IP Ansible uses to reach the host
- `ip_address` + (`mac_addresses` for physical hosts | `mac_address` for VMs) ‚Äî host appears as a dnsmasq static DHCP+DNS reservation on OPNsense. Physical hosts use a LIST of every NIC's MAC (multi-NIC hosts collapse to one OPNsense entry with all MACs in `hardware_addr`, and every entry is also a valid iPXE boot MAC). VMs have exactly one NIC and so a single `mac_address` string, which is also the MAC Tofu provisions on the VM NIC.
- `static_ip: true` ‚Äî suppress the dnsmasq reservation (e.g. the router itself)
- `vm: { proxmox_node, cores, memory, disk_size, tags, ... }` ‚Äî Tofu creates this VM on Proxmox, pinned to `proxmox_node`. VM MACs use the locally-administered `02:` prefix. VM boot disks live on the node-local `vms` PVE storage; no VM ever receives a Longhorn-backed data disk (replicas live only on baremetal kube_workers ‚Äî see Storage carving below).

Groups define topology:
- `router` ‚Üí OPNsense (`opnsense01` at `10.1.1.1`)
- `physical` ‚Üí umbrella for **every** baremetal Linux box (the 17-host tier targets this). Children:
  - `hypervisors` ‚Üí PVE hosts only (the 20-hypervisor tier targets this). Children:
    - `pve_standalone` ‚Üí every PVE host lives here (today: `pve-home-01` only). No corosync, no shared storage.
    - `pve_cluster` ‚Üí **gated off**: empty group kept as deprecated scaffolding only. `20-hypervisor/20-cluster.yml` asserts this group is empty and fails the play otherwise.
  - `workers` ‚Üí plain Linux baremetal (no PVE). k3s baremetal workers, app hosts, etc. Currently: `worker-home-02` (also listed in `kube_workers`). Workers do NOT appear in `proxmox_endpoints` / `template_*_ids` and Tofu does not see them.
- `foundation` ‚Üí top-level group containing the hypervisor(s) that must be all-the-way-up (full stack + bootserv01 LXC + bootserv role) before any other baremetal can come into being. Targeted by the 13-foundation tier via `target_hosts: foundation`. Today: just `pve-home-01`. Members MUST also appear under `hypervisors`.
- `switches` ‚Üí managed network gear that wants a static lease
- `virtual-machines` ‚Üí has children `databases` and `kubernetes`
- `kubernetes` ‚Üí children `kube_control_plane` (kube-ctl-01..03) and `kube_workers` (kube-worker-01; -02/-03 commented out). All VM-based kube nodes are pinned to `pve-home-01` while it is the only hypervisor.
- `containers` ‚Üí `bootserv` (bootserv01 LXC on pve-home-01) + `webservers` placeholder

`all.vars.proxmox_endpoints` is a `{hypervisor ‚Üí https://‚Ä¶:8006}` map (one API endpoint per PVE host). Adding a hypervisor: see `tofu/README.md` ‚Äî extend `proxmox_api_tokens` in `group_vars/all/vars.yml`, then `cp -r tofu/per-node/pve-home-01 tofu/per-node/<new-host>` and edit one identifier in its `main.tf`.

### Provisioning flow

VMs are declared once in `inventory.yaml` (any host with a `vm:` block). Each Proxmox hypervisor has its own Tofu workspace under `tofu/per-node/<host>/` with its own state file. The workspace calls `tofu/modules/hypervisor`, which walks `inventory.yaml`, slices it to guests pinned to that hypervisor, and instantiates the shared `tofu/modules/vm` + `tofu/modules/lxc` modules. The MAC declared in inventory is set on the VM NIC, so the static DHCP reservation pushed to OPNsense in step 1 of the bootstrap (`just do-router-dhcp`) lights up the moment the VM DHCPs.

**State-per-hypervisor**: standalone hypervisors don't share an API and don't share Tofu state. Each per-node directory has a single non-aliased `proxmox` provider and is self-contained ‚Äî adding a hypervisor is `cp -r` + edit one identifier (see `tofu/README.md`). No alias gymnastics, no cross-node coupling at the Terraform layer.

`playbooks/poochella/infra/30-guests/10-opentofu.yml` loops over the `hypervisors` group and per-iteration:
1. Asserts the iterated hypervisor has a token in the vault (`proxmox_api_tokens[node]`) **and** that `tofu-lan` exists on that node via its own `pveum user token list`
2. Runs `community.general.terraform` against `tofu/per-node/<that hypervisor>/`, passing the single per-iteration token via `TF_VAR_proxmox_api_token` (sourced from `proxmox_api_tokens[item]`).

`playbooks/poochella/infra/13-foundation/80-tofu-infra-lxcs.yml` loops over the `foundation` group with the same shape, but with `targets: [module.hypervisor.module.infra_lxcs]` on the `community.general.terraform` call ‚Äî so it creates bootserv01 without incurring the cost (or risk) of materializing the VMs in that hypervisor's workspace. Targeting is per-invocation against the per-node state; the later un-targeted apply reconciles everything (the LXC ends up as a no-op).

VM disks live on the **node-local** `vms` LVM-thin storage (same storage ID on every hypervisor, each backed by that node's own VG). Templates are **produced once on foundation, published to bootserv01, consumed by every hypervisor**: `13-foundation/95-bake-publish-images.yml` runs on the foundation node ‚Äî downloads the pinned upstream Debian cloud image, `virt-customize`s + `virt-sysprep`s the qcow2, snapshots the local LXC template (built by `50-bootstrap-lxc-template.yml`) via `vzdump`, then `pct push`es both artifacts + sha256 sidecars into bootserv01's `/srv/http/cloud-images/` and `/srv/http/lxc-templates/`. Every Proxmox node (including foundation) then runs `20-hypervisor/40-templates.yml` which fetches from `http://bootserv01.lan/...`, verifies sha256, and `qm`/`pct template`s locally ‚Äî peers get a byte-identical copy of foundation's template. Foundation's LXC `pct status` check short-circuits the LXC consume step (the local template from the bootstrap pass satisfies it); foundation does a self-fetch loopback for the VM template (cheap, keeps the consume code path symmetric). The shared image manifest (`vm_templates_base` / `lxc_templates_base`) lives in `playbooks/poochella/infra/vars/templates.yml`, loaded by both the producer and the consumer playbooks. Cloud image URLs are **pinned to dated upstream snapshots** ‚Äî re-pin by bumping `image_url` + `image_file` together; producer sees no match on bootserv01 and rebakes+republishes; consumers see no local VMID/CTID match for the new artifact and re-templatize. Each node's template VMID/CTID comes from the explicit `template_vm_ids` / `template_ct_ids` maps in `inventory.yaml` (single source of truth, also read by `tofu/modules/hypervisor/locals.tf`). Each VM is created on and pinned to its `proxmox_node` and cloned from that node's local template. The bpg/proxmox v0.106 `initialization` block has no `hostname` field ‚Äî VM `name` propagates to cloud-init instead. DNS + search domain reach guests via DHCP options set on OPNsense (single source of truth for DNS).

### Storage carving (DESTRUCTIVE)

**Declared, not discovered ‚Äî but by stable selector, not device name.** Each physical host's disk layout is the `storage.disks` list in `inventory.yaml` (single source of truth; the schema is the SAME for hypervisors and workers ‚Äî the role distinguishes them by which fields each partition declares). Each disk has a `select:` (`boot` | `{model,serial}` | `{min_size_gib|size_gib, rotational}`), an optional per-disk `wipe: refuse|force`, and an ordered `partitions:` list. In OVERWRITE mode a non-`boot` selector **must resolve to exactly one** non-boot, non-removable disk ‚Äî 0 or >1 matches is a **hard preflight failure** (the role never guesses, never auto-claims). In INFO mode the same zero/many path is tolerant and emits `MISSING (disk not found)` for the orphan partitions instead of halting. Selectors match hardware attributes, so kernel renames (`sdX`/`nvmeXnY` shuffle on disk add) are irrelevant and swapping a disk for an identical/larger one needs **no inventory edit** when the selector is attribute-based. `just disk-plan` (read-only `15-storage.yml` with `host_disks_action=info`) reports PRESENT/MISSING per declared partition on every physical host. The paste-ready selector skeleton previously emitted by `disk-plan` is still on disk as `roles/host-disks/tasks/discover.yml` but is no longer wired into any `just` recipe.

Each partition declares `label` (GPT partlabel + idempotency key, **unique per node**), `vg`, `thinpool`, `size` (sgdisk `+SIZE`; `0` = fill), then **EXACTLY ONE** of (XOR, enforced in preflight): `pve_storage` (a **frozen contract**: `tofu/modules/vm` hardcodes `vms`; Tofu does *not* read `storage:` so the schema is Ansible-only) + `content`, OR `mount: { path, fstype, lv_name?, lv_size? }` (the role builds a thin LV, ext4s it, and fstab-mounts it). Several partitions on **different physical disks** may share one `vg`/`thinpool` ‚Äî that is how a NVMe + HDD present as one larger pool (e.g. `longhornthin` mounted at `/var/lib/longhorn` on a baremetal `kube_worker`; the `mount:` block lives on exactly ONE partition per pool). The hypervisor-only convention (asserted in preflight, see `group_vars/hypervisors.yml`): every hypervisor yields `vms` (universal ‚Äî every VM/LXC boot disk + cloud-init drive uses it). That is the only hypervisor-side PVE storage in the contract ‚Äî Longhorn replica state lives on baremetal kube_workers only, never on a hypervisor pool, so a small boot NVMe is sufficient on every hypervisor (pve-home-01 today: 256 GB NVMe, vms only).

`17-host/15-storage.yml` invokes the `host-disks` role, which dispatches via `tasks/main.yml` on `host_disks_action`: `info` runs `preflight.yml` + a read-only `inspect.yml` that never halts on missing partlabels or missing disks; `overwrite` runs `preflight.yml` ‚Üí `carve.yml` (‚Üí `_wipe_disk.yml` + `_carve_one.yml`) ‚Üí `storage_pools.yml` (the destructive path ‚Äî explicit consent); `skip` is a single `meta: end_host`. Preflight resolves every selector to a device and (overwrite mode only) asserts the partlabel-uniqueness and pool-target-consistency invariants. Hypervisor-only preflight steps (gated on overwrite mode AND on `inventory_hostname in groups['hypervisors']`): the `qm/pct list` + template detection guards, plus the frozen-contract assertion that this hypervisor yields `vms`. Overwrite mode is "nuke from orbit": preflight stops every running VM/CT (`qm/pct stop`, 60s graceful) then `qm/pct destroy --purge`es every guest on this hypervisor ‚Äî real and template ‚Äî so the carve can rebuild the storage they sat on. Real guests come back via the un-targeted `tofu apply` in `30-guests/10-opentofu.yml`; regenerable templates come back via `20-hypervisor/40-templates.yml` (consume from bootserv01). If the foundation node itself is being carved, that pass also wipes the bootstrap LXC template ‚Äî the next `13-foundation/site.yml` run rebakes it via `50-bootstrap-lxc-template.yml` and re-publishes via `95-bake-publish-images.yml`. The legacy `carve_destroy_templates` flag and the real-guest hard refusal are both gone ‚Äî `host_disks_action: overwrite` IS the consent. Carve runs a per-disk **wipe gate** once per disk (boot disk never wiped; a disk already carrying our partlabels is skipped ‚Äî idempotent; a disk with **foreign signatures** is REFUSED unless it declares `wipe: force`), then carves each partition in declared order (idempotent on the GPT partlabel). `storage_pools.yml` builds the `vg`/`thinpool` LVM-thin pool (a VG may span multiple PVs), **grows** an existing thin pool onto a newly-fitted larger disk (`lvextend +100%FREE`), and ‚Äî for any partition with `mount:` ‚Äî creates the thin LV, ext4-formats it, and persists the mount in fstab. The PVE-side publish step is decoupled: `20-hypervisor/30-pve-storage-register.yml` runs `pvesm add lvmthin` for each unique `pve_storage`, scoped `--nodes <self>` (independent nodes ‚áí per-node `/etc/pve/storage.cfg`). Disk swap / capacity / NAS-backup runbook: `docs/storage-disk-runbook.md`. **Data safety is a Longhorn-layer property (separate Flux repo), not an LVM one** ‚Äî the carve only makes the empty re-init safe; Longhorn's replica-2 + zone anti-affinity is what survives a disk pull.

Longhorn placement: replica state lives ONLY on baremetal `kube_workers` (today: `worker-home-02`). Each baremetal worker declares `storage.disks` with a `mount: /var/lib/longhorn ext4` partition that `host-disks` realises directly from `17-host/15-storage.yml` ‚Äî no VM ever receives a Longhorn-backed data disk, and no hypervisor pool backs Longhorn. `40-kube/10-longhorn-host-prep.yml` runs cluster-wide and installs ONLY the consumer prereqs (open-iscsi + nfs-common + iscsid) so the schedulable VM control planes can attach + mount Longhorn volumes over iSCSI/NFS. The Flux-managed Longhorn install must restrict storage scheduling to the workers (`createDefaultDiskLabeledNodes` + the `node.longhorn.io/create-default-disk=true` label set on `kube_workers` only) so replica + rebuild IO stays on the baremetal workers and never lands on the etcd CP VMs.

### k3s + kube-vip

`playbooks/poochella/infra/40-kube/20-k3s.yml` builds an embedded-etcd HA cluster (k3s runs inside VM-based control planes, plus an agent on the baremetal `worker-home-02`; CP VMs are all pinned to `pve-home-01` while it is the only hypervisor, so that host is the etcd blast radius):
1. First CP runs `k3s server --cluster-init`; kube-vip is then deployed only on that node to bring up the L2 VIP at `k3s_kube_vip_address` (currently `10.1.1.50`)
2. Remaining CPs join `serial: 1` via `https://<api-endpoint>:6443` where api-endpoint is `k3s_api_dns_name` if set, else the VIP; workers join after
3. kubeconfig is fetched back to `{{ playbook_dir }}/../../../kubeconfig` (repo root) with its `server:` URL rewritten to the DNS name (else the VIP) ‚Äî `export KUBECONFIG=$(pwd)/kubeconfig` to use it

The VIP must be a free address on the same L2 segment as the control planes. Config lives in `group_vars/kubernetes.yml`. Layered on top of the VIP is `k3s_api_dns_name: api.doghouse.lan` ‚Äî pushed to OPNsense Unbound by `10-router/30-unbound.yml` (the more-specific override coexists with the `*.doghouse.lan` ingress wildcard). The k3s role uses the DNS name for the joiner `--server` URL, the rewritten kubeconfig `server:`, and as a TLS SAN on the apiserver cert; kube-vip still owns sub-second L2 failover between CPs, the DNS layer is sugar on top. Renumbering the VIP is therefore a one-line DNS edit. The bootstrap order (10-router before 40-kube) already satisfies the soft DNS dependency the layer introduces.

### Roles

Roles under `roles/` are units of work, not "things to install":
- `host-base` ‚Äî every physical-host baseline: install sudo+zsh, import `apt-no-auto-upgrades`, create `ansible`/`pani`/`tourmanager` users via `create-user`, set root password from vault. Driven by `17-host/10-bootstrap.yml`. The user policy is in `defaults/main.yml`'s `host_base_users`.
- `ssh-hardening` ‚Äî drops `/etc/ssh/sshd_config.d/10-hardening.conf` from a template: disables password+root login globally, but a `Match User <breakglass>` block keeps password auth for one specific user (defaults to `tourmanager`). Runs after `host-base` in the 17-host tier so the keys are enrolled before password auth turns off. Asserts the break-glass user exists; the drop-in is validated with `sshd -t` before it lands.
- `hypervisor` ‚Äî PVE-only repo flip (enterprise ‚Üí no-subscription PVE+Ceph). Its `cluster_ssh.yml` / `cluster_create.yml` / `cluster_join.yml` task files are **gated off** by an assertion in `20-hypervisor/20-cluster.yml` and kept on disk as deprecated reference. Driven by `20-hypervisor/10-bootstrap.yml`. Generic baremetal setup that used to live inline here is now in `host-base`.

## Conventions

- Playbook filenames are prefixed `NN-` to encode run order within `infra/`. Don't reorder by renumbering unless you also understand the dependencies.
- The Ansible remote user is `ansible` with key `~/.ssh/ansible` (set in `ansible.cfg`). Cloud-init bakes this user into VM templates (the producer step `13-foundation/95-bake-publish-images.yml`); the `host-base` role creates it on every physical host.
- Pre-conditions are documented as comment blocks at the top of each infra playbook ‚Äî read those before editing.
- `playbooks/wip/` is scratchpad / in-progress work, not part of `site.yml`.

## Collaboration rules (for Claude)

- **Diagnose by running read-only ansible ad-hoc commands directly**, e.g. `ansible pve-home-01 -m shell -a 'pveam list local' --become`, `ansible bootserv01 -m shell -a 'systemctl is-active dnsmasq nginx' --become`. Do not ask the user to copy/paste terminal output when an ad-hoc command can fetch the same information. The `ansible` user + `~/.ssh/ansible` key already trusts every fleet host.
- **Never run a destructive command on the fleet without explicit per-command user permission.** "Destructive" = anything that mutates state on the host (`pct destroy`, `pveam remove`, `pvesm remove`, `rm`, `systemctl stop/disable`, `apt remove`, `qm destroy`, partitioning / disk wipes, killing processes, writing files outside `/tmp`, etc.). Reading config, listing resources, dumping logs, and `--check`/`--diff` dry-runs are fine. The same rule applies to OpenTofu (`tofu apply` against existing resources is destructive in principle) and to anything that touches OPNsense state.
- **Use the NATO alphabet when naming cluster entities. Always start at Alpha move up.**

## bootserv01 / iPXE boot flow

`bootserv01` is an LXC on `pve-home-01` (inventory `containers.bootserv.bootserv01`, `lxc:` block tagged `infra`) that serves TFTP (iPXE chainload binaries) and HTTP (boot scripts + Debian netboot kernel/initrd + per-host preseeds) to baremetal hosts that net-boot from their NIC. OPNsense's dnsmasq advertises `ipxe.efi` / `undionly.kpxe` to vanilla-PXE clients (chainload) and a script URL to user-class iPXE clients. Per-host iPXE dispatch + preseed are rendered by the `bootserv` Ansible role from every inventory host whose `storage.disks` carries a `select: boot` entry with an `hw: {model, serial}` pin (the marker that the host is installable via this preseed). The dispatcher matches against every entry in that host's `mac_addresses` list ‚Äî there is no separate "boot MAC" field.

**Brought up by the 13-foundation tier.** The bootserv01 LXC is created by `13-foundation/80-tofu-infra-lxcs.yml` (loops over the `foundation` group and runs `tofu apply -target=module.hypervisor.module.infra_lxcs` against each foundation node's per-node workspace ‚Äî does NOT touch any VM) and configured by `13-foundation/90-bootserv.yml` (was `30-guests/15-bootserv.yml`). Foundation must complete before any other baremetal can be netbooted ‚Äî that's the whole point of the tier.

**Tofu ordering invariant (still relevant for the un-targeted per-node apply in 30-guests)**: LXCs whose `lxc.tags` contains `infra` come up before any VMs. In `tofu/modules/hypervisor/main.tf`, `module.vms` declares `depends_on = [module.infra_lxcs]`, so even when the un-targeted apply runs first on a fresh fleet (no foundation tier run beforehand), bootserv01 still comes up before kube/dev VMs on the same hypervisor.
